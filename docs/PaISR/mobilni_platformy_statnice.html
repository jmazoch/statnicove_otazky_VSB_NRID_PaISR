1. Charakteristiky OS Android

Android je open-source operační systém založený na Linux kernelu, optimalizovaný pro nízkou spotřebu, omezené zdroje a mobilní hardware. Android není totéž jako klasická desktopová distribuce Linuxu.

Klíčové odlišnosti od klasického Linuxu:

  • používá úsporný scheduler
  • má vlastní systém řízení aktivit, ne procesů
  • každá aplikace běží v sandboxu, odděleně od ostatních
  • všechny aplikace jsou psané v bytecode (.dex), který běží v ART (Android Runtime)

Android Runtime (ART)

ART je výkonný engine, který:

  • načítá DEX bytecode
  • provádí JIT (just-in-time) i AOT (ahead-of-time) kompilaci
  • používá garbage collector a optimalizovanou správu paměti
  • zajišťuje kompatibilitu mezi API verzemi

Architektura Androidu — vrstvy

Čtyřvrstvá architektura Androidu — od Linux kernelu přes HAL a ART/Framework až k samotným aplikacím.

HAL — proč existuje: Android používá HAL (Hardware Abstraction Layer), aby sjednotil chování hardwaru — aplikace neřeší rozdíly mezi výrobci. Každý výrobce píše svůj HAL ovladač, ale API pro aplikace zůstává stejné.

2. API Levely a kompatibilita verzí

Každá verze Androidu má přidělen API level — číslo, které určuje, jaké funkce a třídy jsou dostupné. Vývojář nastavuje tři klíčová čísla, která řídí kompatibilitu aplikace.

Tři klíčová čísla v projektu

minSdkVersion

Minimální verze OS, na které aplikace poběží.
Např. minSdk 21 = od Android 5 výše.

Musí být vždy nejnovější.

AndroidX knihovny nahrazují staré Support Library a přidávají zpětnou kompatibilitu — umožňují používat moderní API i na starších verzích Androidu.

3. Životní cyklus aktivity

Aktivita = jedna obrazovka aplikace, tvořená layoutem (XML) a logikou (Java/Kotlin). Android ji řídí pomocí přesně definované sady callback metod — životního cyklu.

Metody životního cyklu — pořadí

  1. onCreate() — vytvoření UI, inicializace, setContentView()
  2. onStart() — aktivita je viditelná, ale ještě ne v popředí
  3. onResume() — aktivita je v popředí, interaktivní
  4. onPause() — částečně zakrytá (dialog, jiná aktivita)
  5. onStop() — aktivita není vidět
  6. onDestroy() — vyčištění, ukončení
  7. onRestart() — návrat z pozadí (onStop → onRestart → onStart)

Rotační sekvence — extrémně důležitá (učitel se ptá!)

Při otočení obrazovky proběhne přesně toto pořadí:

  1. onPause()
  2. onSaveInstanceState() ← uložení stavu před zničením
  3. onStop()
  4. onDestroy() ← aktivita je zničena!
  5. onCreate() ← nová instance
  6. onStart()
  7. onRestoreInstanceState() ← obnovení stavu
  8. onResume()
Proč je rotace destruktivní: Android při rotaci aktivitu kompletně znovu vytvoří. Pokud si data neulož v onSaveInstanceState(), po rotaci jsou ztracena. Moderní řešení: ViewModel přežije rotaci bez ztráty dat.

5. Vestavěné senzory — IMU, tlak, teplota, osvětlení

Android Sensor API umožňuje přístup k fyzickým senzorům zařízení přes standardizované rozhraní SensorManager + SensorEventListener, bez závislosti na konkrétním HW výrobci.

Android Sensor API

SensorManager sensorManager = (SensorManager) getSystemService(SENSOR_SERVICE);
Sensor accel = sensorManager.getDefaultSensor(Sensor.TYPE_ACCELEROMETER);

sensorManager.registerListener(new SensorEventListener() {
    @Override
    public void onSensorChanged(SensorEvent event) {
        float x = event.values[0];  // m/s²
        float y = event.values[1];
        float z = event.values[2];
    }
    @Override
    public void onAccuracyChanged(Sensor sensor, int accuracy) {}
}, accel, SensorManager.SENSOR_DELAY_NORMAL);

Typy senzorů a jejich vlastnosti

SenzorMěříProblém / poznámkaPrůmyslové využití
Akcelerometr Zrychlení X/Y/Z [m/s²], cítí gravitaci → naklonění Odstředivá síla „mate" výsledky při rotaci Detekce vibrací, nárazů, pádu, naklonění
Gyroskop Úhlová rychlost [rad/s] Drift při integraci — nutná kompenzace Stabilní měření orientace, navigace, robotika
Magnetometr Zemské magnetické pole [μT] Citlivý na kovové předměty v okolí Světové strany, magnetické anomálie
Barometr (tlak) Atmosférický tlak [hPa] Teplotní drift Výška nad mořem, HVAC monitoring, změna tlaku
Teplotní senzor Teplota zařízení [°C] Měří teplotu uvnitř telefonu, ne okolí Monitoring přehřívání, teplotní kompenzace
Světelný senzor Osvětlení [lux] — Monitoring osvětlení prostředí, úspora energie

IMU — Inerciální měřicí jednotka

IMU kombinuje akcelerometr + gyroskop (+ magnetometr). Pro průmyslové aplikace se fúzují data ze všech tří senzorů pomocí Kalmanového filtru nebo komplementárního filtru — výsledkem je stabilní odhad orientace a polohy bez kumulace chyb.

Průmyslové využití senzorů

• Diagnostika vibrací motorů
• Prediktivní údržba (predictive maintenance)
• Sledování orientace robotických ramen
• Monitoring prostředí (teplota, tlak, světlo)
• Detekce kolizí a abnormálních stavů

SensorManager — frekvence měření

SENSOR_DELAY_FASTEST — max. rychlost (~1 ms)
SENSOR_DELAY_GAME — ~20 ms
SENSOR_DELAY_UI — ~60 ms
SENSOR_DELAY_NORMAL — ~200 ms

Pro průmyslové vibrace volíme FASTEST.

Akcelerometr vs. gyroskop: Akcelerometr cítí gravitaci i pohyb — při rotaci ho mate odstředivá síla. Gyroskop měří úhlovou rychlost přesně, ale integrací se hromadí drift. Ideální kombinace pro průmyslové aplikace je fúze obou senzorů.

6. Fotoaparát — Camera2 API a průmyslové zpracování obrazu

Camera2 API umožňuje nízkoúrovňové řízení fotoaparátu — expozice, ISO, clona, fokus, sekvenční snímky. Pro průmyslové aplikace je to základ inspekce kvality, OCR, detekce vad nebo termodiagnostiky.

Průmyslové aplikace obrazu

  • Inspekce kvality — automatická detekce vad na výrobní lince
  • OCR — čtení čárových kódů, QR kódů, štítků
  • Detekce vad — praskliny, geometrie, barva produktu
  • Měření rozměrů — kalibrovaný kamerový systém
  • Termodiagnostika — pokud je k dispozici termokamera

Camera2 API — co lze řídit

Parametry snímání

• ISO (citlivost)
• Expozice (čas závěrky)
• Clona (f-stop)
• Fokus (manuální / AF)
• Sekvenční snímky (burst)

Formáty a zpracování

• JPEG (komprimovaný výstup)
• RAW (surová data ze senzoru)
• YUV (pro real-time zpracování)
• BMP (bitová mapa)
• Převod do šedi: 0.30R + 0.59G + 0.11B

Digitální zpracování obrazu

  • Konverze do šedi: Y = 0.30·R + 0.59·G + 0.11·B
  • Filtrace: blur (rozmazání šumu), sharpen (zvýraznění hran)
  • FFT pro spektrální analýzu textur a vibrací zobrazených v obraze
  • Prahování (thresholding) pro binární segmentaci
// Camera2 API – základní struktura
CameraManager manager = (CameraManager) getSystemService(CAMERA_SERVICE);
manager.openCamera(cameraId, new CameraDevice.StateCallback() {
    @Override
    public void onOpened(CameraDevice camera) {
        // Nastavení parametrů a spuštění preview
        CaptureRequest.Builder builder =
            camera.createCaptureRequest(CameraDevice.TEMPLATE_PREVIEW);
        builder.set(CaptureRequest.SENSOR_SENSITIVITY, 800);   // ISO
        builder.set(CaptureRequest.CONTROL_AF_MODE,
            CaptureRequest.CONTROL_AF_MODE_AUTO);
    }
}, null);

7. Mikrofon — AudioRecord API, PCM, FFT analýza

Mikrofon mobilního zařízení je plnohodnotný senzor pro průmyslové akustické analýzy. Přes AudioRecord API lze získávat PCM vzorky v reálném čase a provádět FFT analýzu vibrací, monitorování ložisek nebo detekci anomálií zvuku.

Základy zvuku

  • Slyšitelné pásmo: 16 – 20 000 Hz
  • Rychlost zvuku: 340 m/s
  • Intenzita se vyjadřuje v dB (dekadický logaritmus)
  • Křivky stejné hlasitosti (izofony) — různé frekvence jsou vnímány různě hlasitě při stejném tlaku

Typy mikrofonů

TypPrincipPoužití
ElektrodynamickýMembrána v magnetickém poli → indukuje napětíStudiové mikrofony
ElektretovýTrvale nabitá folie → piezoelektrický efektTelefony, náhlavní soupravy
KondenzátorovýVariabilní kapacita mezi membránou a elektrodouStudijní záznamy, měřicí mikrofony
PiezoelektrickýTlak → elektrické napětíPrůmyslové senzory vibrací

AudioRecord API — PCM vzorkování

int sampleRate = 44100;  // Hz
int bufferSize = AudioRecord.getMinBufferSize(
    sampleRate,
    AudioFormat.CHANNEL_IN_MONO,
    AudioFormat.ENCODING_PCM_16BIT);

AudioRecord recorder = new AudioRecord(
    MediaRecorder.AudioSource.MIC,
    sampleRate,
    AudioFormat.CHANNEL_IN_MONO,
    AudioFormat.ENCODING_PCM_16BIT,
    bufferSize);

recorder.startRecording();
short[] buffer = new short[bufferSize];
recorder.read(buffer, 0, bufferSize);  // PCM vzorky
// → následuje FFT analýza

PCM digitalizace zvuku

  • PCM_16BIT — 16bitová hloubka (65 536 úrovní), standardní
  • PCM_FLOAT — 32bitová plovoucí desetinná čárka, vyšší přesnost
  • Vzorkovací frekvence: typicky 44 100 Hz nebo 48 000 Hz

Průmyslové využití mikrofonu

Monitoring strojů pomocí FFT

• Monitoring ložisek (charakteristické frekvence závad)
• Diagnostika ventilátorů, motorů, čerpadel
• Detekce kavitace, rezonancí
• Prediktivní údržba — trend ve spektru

Streamování dat

• Real-time FFT přímo v telefonu
• Streaming přes TCP (persistentní stream)
• Streaming přes HTTP (REST API)
• Streamování přes MQTT (IoT broker)

8. Komunikační protokoly — TCP, HTTP/S, MQTT, Bluetooth

Mobilní platforma komunikuje s průmyslovými systémy přes standardní protokoly. Volba protokolu závisí na požadavcích na spolehlivost, latenci, šířku pásma a energetickou náročnost.

8.1 TCP

  • Spolehlivý — garantuje doručení a pořadí paketů
  • Persistentní spojení server ↔ klient
  • Typické scénáře: řízení robotů, datové brány mezi PLC a aplikací, real-time měření, telemetrie
// TCP klient v Androidu (v coroutine nebo vláknu)
Socket socket = new Socket("192.168.1.100", 8080);
PrintWriter out = new PrintWriter(socket.getOutputStream(), true);
BufferedReader in = new BufferedReader(
    new InputStreamReader(socket.getInputStream()));
out.println("START_MEASUREMENT");
String response = in.readLine();

8.2 HTTP / HTTPS

  • Aplikační protokol — model request → response
  • REST API s JSON nebo XML payloadem
  • HTTPS = HTTP + TLS šifrování
  • Průmyslové použití: dashboardy, cloud monitoring, SCADA webové rozhraní
// HTTP GET request v Androidu (OkHttp)
OkHttpClient client = new OkHttpClient();
Request request = new Request.Builder()
    .url("https://api.factory.cz/sensor/temperature")
    .build();
Response response = client.newCall(request).execute();
String json = response.body().string();

8.3 MQTT

MQTT je lightweight publish/subscribe protokol navržený pro IoT a M2M komunikaci s minimální spotřebou energie a šířky pásma.

Architektura MQTT

• Broker = centrální uzel (např. Mosquitto, HiveMQ)
• Publisher = posílá data na topic
• Subscriber = odebírá data z topicu
• Topic: hierarchická struktura (factory/line1/temp)

QoS úrovně

QoS 0 — fire and forget, nejrychlejší, bez záruky
QoS 1 — doručeno alespoň jednou, možné duplikáty
QoS 2 — doručeno přesně jednou, nejpomalejší

Pro průmysl: QoS 1 nebo 2.

MQTT je perfektní pro: senzory s omezenou energií, prediktivní údržbu, IoT gateway, datové sběry z výrobní linky.

8.4 Bluetooth / Bluetooth Low Energy (BLE)

TypProfilPropustnostPrůmyslové využití
Klasický BT SPP (Serial Port Profile) = virtuální COM port Vyšší (až 2 Mbps) Jednoduché senzory, řídicí aplikace, diagnostika strojů
BLE (Bluetooth Low Energy) GATT — Services → Characteristics → Descriptors Nízká (ale dostačující pro senzory) Wearables, IoT senzory, průmyslové tagy, asset tracking

GATT hierarchie (BLE)

Service — logická skupina
Characteristic — datová hodnota
Descriptor — metadata charakteristiky
Vše identifikováno přes UUID.

BLE výhody v průmyslu

• Nízká spotřeba energie
• Dosah 10–100 m
• Stovky zařízení najednou
• Beacon technologie

BLE nevýhody

• Nízká propustnost
• Vyšší latence než WiFi
• Interference při velkém počtu zařízení
• Párování nutné

9. Kyberbezpečnost mobilních platforem

Android má dvoustupňový model oprávnění — statická deklarace v manifestu a dynamické potvrzení uživatelem za běhu. Citlivá data se ukládají do šifrovaného úložiště, komunikace probíhá přes HTTPS s certificate pinningem.

9.1 Oprávnění (Permissions)

Manifest permissions (statická)

Deklarují se v AndroidManifest.xml — instalační čas.
Např.:
<uses-permission android:name=
"android.permission.INTERNET"/>

Runtime permissions (dynamická)

Uživatel musí potvrdit za běhu aplikace:
• Kamera
• Mikrofon
• Poloha (GPS)
• Úložiště
• Bluetooth

// Kontrola a žádost o runtime permission
if (ContextCompat.checkSelfPermission(this, Manifest.permission.CAMERA)
        != PackageManager.PERMISSION_GRANTED) {
    ActivityCompat.requestPermissions(this,
        new String[]{Manifest.permission.CAMERA},
        REQUEST_CODE_CAMERA);
}

9.2 Šifrování dat

  • Android podporuje File-Based Encryption na úrovni OS
  • Citlivá data ukládat do EncryptedSharedPreferences
  • Privátní klíče ukládat do Android KeyStore (RSA, AES) — klíče neopouštějí HW čip
  • Komunikace přes HTTPS + certificate pinning (ověření konkrétního certifikátu serveru)

10. Co umět říct u státnic

  • Android Open-source OS na Linux kernelu. Každá aplikace v sandboxu, vlastní UID, bytecode DEX běží v ART (JIT + AOT).
  • ART načítá DEX bytecode, provádí JIT i AOT kompilaci, spravuje paměť přes garbage collector.
  • API level = číslo verze Androidu. Tři klíčová čísla: minSdkVersion, targetSdkVersion, compileSdkVersion.
  • Životní cyklus onCreate → onStart → onResume → onPause → onStop → onDestroy. onRestart při návratu z pozadí.
  • Rotace! onPause → onSaveInstanceState → onStop → onDestroy → onCreate → onStart → onRestoreInstanceState → onResume. Toto pořadí se učitel ptá!
  • Kotlin doporučený jazyk — null safety, coroutines, méně verbose než Java.
  • Flutter multiplatformní, Skia renderer, widgety, větší APK.
  • Akcelerometr zrychlení X/Y/Z, cítí gravitaci. Gyroskop úhlová rychlost. IMU = fúze obou přes Kalmanův filtr.
  • Průmysl senzory diagnostika vibrací, prediktivní údržba, monitoring prostředí.
  • Camera2 API řídí ISO, expozici, clonu, fokus. Průmysl: inspekce kvality, OCR, detekce vad. Převod do šedi: 0.30R + 0.59G + 0.11B.
  • AudioRecord API PCM_16BIT nebo PCM_FLOAT. FFT → spektrální analýza → monitoring ložisek, motorů. Streaming přes TCP/MQTT.
  • TCP spolehlivý, garantuje pořadí, persistentní spojení. Pro telemetrii a real-time měření.
  • MQTT publish/subscribe, broker (Mosquitto), QoS 0/1/2. Perfektní pro IoT senzory a průmyslové sběry dat.
  • BLE GATT Services → Characteristics → Descriptors. Nízká spotřeba, nízká propustnost — stačí pro senzory.
  • Permissions Manifest = statická deklarace. Runtime = uživatel potvrzuje kamera/mikrofon/polohu za běhu.
  • KeyStore = nejbezpečnější úložiště pro klíče. External Storage = nebezpečné.
  • Publikace podepsání (keystore) → .AAB → Google Play Console → automatické skenování → schválení.
