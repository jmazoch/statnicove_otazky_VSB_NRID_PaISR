1. Základní vlastnosti FreeRTOSu

FreeRTOS je minimalistický operační systém reálného času určený primárně pro mikrokontroléry. Poskytuje scheduler, úlohy ve formě vláken, synchronizaci, komunikaci mezi úlohami a časovou základnu.
  • je dostupný zdarma a je navržen pro embedded systémy,
  • má minimalistický přístup a statickou konfiguraci,
  • obsahuje plnohodnotný plánovač úloh,
  • úlohy jsou vlákna, ne procesy,
  • nepoužívá klasickou procesovou izolaci jako Linux nebo Windows,
  • interakce mezi vlákny zahrnuje synchronizaci a výměnu dat,
  • operační systém má koncepci času, tedy existuje časová základna.
Dobrá věta ke zkoušce: FreeRTOS není „malý Linux“. Je to přesný dispečer v mikrokontroléru, který rozhoduje, která úloha dostane CPU, kdy se zablokuje a jak se probudí.

2. Konfigurace FreeRTOSu

Konfigurace se provádí v souboru FreeRTOSConfig.h. Z hlediska otázky jsou nejdůležitější volby pro scheduler, časovou základnu, idle hook, časovače a queue sets.

VolbaVýznamTypická poznámka
configUSE_PREEMPTIONZapíná preemptivní scheduler.1 = preemptivní, 0 = kooperativní.
configUSE_TIME_SLICINGZapíná časové dělení mezi úlohami stejné priority.Má význam jen u preemptivního scheduleru.
configTICK_RATE_HZFrekvence ticku časové základny OS.Např. 1000 Hz znamená 1 ms tick.
configUSE_IDLE_HOOKPovoluje vApplicationIdleHook().Volá se z idle úlohy.
configUSE_TIMERSPovoluje softwarové časovače.Vyžaduje konfiguraci timer service tasku.
configTIMER_TASK_PRIORITYPriorita daemon/timer service tasku.Ovlivňuje obsluhu timer callbacků.
configTIMER_QUEUE_LENGTHDélka příkazové fronty časovačů.Příkazy timer API se posílají daemon úloze přes frontu.
configTIMER_TASK_STACK_DEPTHVelikost zásobníku timer service tasku.Ve 32bitových slovech, ne v bajtech.
configUSE_QUEUE_SETSPovoluje množiny front.Bez toho nelze použít queue sets.

3. Úlohy, stavy úloh a jejich vytvoření

Úloha ve FreeRTOSu odpovídá vláknu. Každá úloha má vlastní zásobník, vlastní TCB, prioritu a nachází se právě v jednom ze čtyř stavů: running, ready, blocked nebo suspended.

Obr. 1 – Stavy úlohy ve FreeRTOSu a přechody mezi nimi.

StavVýznamTypický přechod
RunningÚloha právě běží na CPU.Scheduler jí přidělil procesor.
ReadyÚloha je připravena běžet, ale CPU má jiná úloha.Čeká, až ji scheduler vybere.
BlockedÚloha čeká na čas, event, semafor, frontu nebo jinou událost.vTaskDelay(), čekání na queue/semafor/event.
SuspendedÚloha je explicitně pozastavena.vTaskSuspend(), návrat přes vTaskResume().

Implementace úlohy

Úloha je C funkce s návratovým typem void a parametrem typu void *. Parametr lze použít pro předání ukazatele na strukturu, číslo, konfiguraci nebo handle.

void TaskFunction(void *pvParameters)
{
    for (;;)
    {
        // tělo úlohy
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}

Vytvoření úlohy

xTaskCreate(
    TaskFunction,        // funkce úlohy
    "TaskName",          // jméno pro ladění
    stackSize,           // velikost stacku, typicky ve slovech
    pvParameters,        // parametr předaný úloze
    priority,            // priorita, vyšší číslo = vyšší priorita
    &taskHandle          // handle, může být NULL
);
  • xTaskCreate() lze volat před spuštěním scheduleru i za běhu programu.
  • vTaskStartScheduler() spustí scheduler. Normálně se z této funkce nevrací, pokud nedojde k chybě.
  • Každá úloha má prioritu v rozsahu 0 až configMAX_PRIORITIES - 1.
  • Idle task má nejnižší prioritu a běží, když jsou všechny uživatelské úlohy blokované.

4. Scheduler: kooperativní a preemptivní plánování

Scheduler přiděluje přístup k CPU jednotlivým úlohám a realizuje přechody mezi stavy úloh. Vybírá pouze z úloh ve stavu Ready.

Kooperativní scheduler

Běžící úloha se musí sama vzdát procesoru, například přes taskYIELD(), vTaskDelay() nebo blokující API.

Preemptivní scheduler

CPU může být běžící úloze odebráno bez jejího souhlasu, například když se probudí úloha s vyšší prioritou nebo při časovém dělení.

VlastnostKooperativníPreemptivní
Kdo rozhoduje o opuštění CPU?Sama běžící úloha.Scheduler podle priority, událostí a politiky plánování.
RizikoJedna špatně napsaná úloha může držet CPU příliš dlouho.Nutná opatrnost u sdílených prostředků a race conditions.
KonfiguraceconfigUSE_PREEMPTION = 0configUSE_PREEMPTION = 1
Chyták: kooperativní scheduler neznamená, že CPU nemůže být přerušeno hardwarovým přerušením. Přerušení procesoru a přepnutí úlohy schedulerem jsou dvě různé věci.

5. FIFO, Round Robin a priority

FreeRTOS nejdříve řeší priority. Teprve pokud existuje více ready úloh se stejnou nejvyšší prioritou, uplatní se konkrétní plánovací strategie, tedy FIFO nebo Round Robin.

Najdi nejvyšší prioritu s Ready tasky
→
Ignoruj nižší priority
→
Mezi stejnou prioritou použij FIFO/RR
StrategieKdy běžící úloha ztrácí CPUKonfigurace
FIFOKdyž přejde do Blocked, skončí, zavolá yield, nebo se objeví ready úloha s vyšší prioritou.configUSE_PREEMPTION = 1
configUSE_TIME_SLICING = 0
Round RobinStejné jako FIFO, ale navíc po vyčerpání časové dotace.configUSE_PREEMPTION = 1
configUSE_TIME_SLICING = 1
Kooperativní režimAž při dobrovolném předání CPU nebo blokování.configUSE_PREEMPTION = 0
configUSE_TIME_SLICING nemá význam.
Nejdůležitější věta: FIFO i Round Robin ve FreeRTOSu řeší hlavně úlohy se stejnou prioritou. Vyšší priorita má absolutní přednost.

Idle úloha

Pokud jsou všechny uživatelské úlohy ve stavu Blocked, procesor nemůže přestat vykonávat instrukce. Běží proto idle task. Ten může volat uživatelskou funkci:

void vApplicationIdleHook(void)
{
    // nízkoprioritní obsluha, úsporný režim, diagnostika...
}

6. Event groups

Event group je synchronizační mechanismus, který předává informaci typu „něco se stalo“. Nepřenáší data, pouze stavové příznaky reprezentované bity.
  • Na 32bitové platformě obsahuje event group 24 uživatelských eventů.
  • Eventy jsou identifikovány bitovou maskou.
  • Více eventů může být nastaveno najednou a nastavení bitů je neblokující operace.
  • Úloha může čekat na jeden event, libovolný z vybraných eventů, nebo všechny vybrané eventy.
  • Čekání může skončit také timeoutem.
  • Eventy, které úlohu probudily, lze automaticky vymazat.
FunkceVýznam
xEventGroupCreate()Vytvoří event group a vrátí handle.
xEventGroupSetBits()Nastaví vybrané event bity.
xEventGroupWaitBits()Čeká na eventy podle masky, timeoutu a AND/OR pravidla.
xEventGroupGetBits()Přečte aktuální stav eventů.
xEventGroupClearBits()Vymaže vybrané event bity.
Praxe: event group je dobrá pro stavové signály typu „data připravena“, „chyba“, „hotová inicializace“. Není dobrá pro přenos hodnot. Event group je cedulka, ne balík.

7. Queues a queue sets

Přenos dat hodnotou a odkazem

U větších datových struktur je neefektivní kopírovat celé bloky dat. Proto se často preferuje předávání odkazem, tedy zero-copy přístup.

ZpůsobPrincipPoznámka
HodnotouDo fronty se kopíruje samotná hodnota nebo struktura.Jednoduché, ale drahé pro velká data.
OdkazemDo fronty se vloží ukazatel na data uložená v RAM.Rychlé, ale musí se hlídat životnost paměti.

Queues

Fronty ve FreeRTOSu slouží k předávání dat mezi úlohami nebo mezi ISR a úlohou. Data jsou čtena v pořadí FIFO.
  • fronta je vytvořena funkcí xQueueCreate(),
  • data se zapisují funkcemi xQueueSend() nebo xQueueSendToFront(),
  • data se čtou funkcemi xQueueReceive() nebo xQueuePeek(),
  • xQueuePeek() přečte položku z čela fronty, ale neodstraní ji,
  • při plné frontě rozhoduje parametr xTicksToWait: nečekat, čekat omezeně, nebo čekat neomezeně.
QueueHandle_t q = xQueueCreate(10, sizeof(uint8_t));

uint8_t value = 42;
xQueueSend(q, &value, pdMS_TO_TICKS(10));

uint8_t received;
xQueueReceive(q, &received, portMAX_DELAY);

Queue sets

Queue set umožňuje jedné úloze čekat na více front zároveň. Pokud se v některé členské frontě objeví data, xQueueSelectFromSet() vrátí handle fronty, ze které má příjemce číst.

FunkceVýznam
xQueueCreateSet()Vytvoří množinu front.
xQueueAddToSet()Přidá frontu do množiny.
xQueueSelectFromSet()Čeká, dokud se data neobjeví v některé členské frontě.
Queue set je odpověď na otázku: „Může jedna úloha čekat na více front zároveň?“ Ano, ale musí být povoleno configUSE_QUEUE_SETS = 1.

8. ISR a výjimky ve FreeRTOSu

Z obslužné rutiny přerušení lze volat jen ty funkce FreeRTOS API, jejichž název končí FromISR. Handler se nesmí chovat blokujícím způsobem.
  • Jde o hardwarové výjimky ARM Cortex-M, tedy systémové výjimky a přerušení, ne C++/Java výjimky.
  • Funkce volaná z handleru neběží v kontextu žádné konkrétní úlohy.
  • Pokud ISR odblokuje úlohu s vyšší prioritou, nastaví se proměnná typu BaseType_t na pdTRUE.
  • Na konci handleru se použije portYIELD_FROM_ISR(), aby mohl kernel po návratu z ISR přepnout kontext.
void some_handler(void)
{
    BaseType_t wake_req = pdFALSE;

    xEventGroupSetBitsFromISR(events, BIT_0, &wake_req);
    xSemaphoreGiveFromISR(semaphore, &wake_req);
    xQueueSendFromISR(queue, &data, &wake_req);

    portYIELD_FROM_ISR(wake_req);
}
Chyták: z ISR nikdy nevolej běžné blokující varianty API. ISR není task a nemá čekat ve frontě jak student na studijním.

9. Kritická sekce, semafory a mutexy

Kritická sekce

Kritická sekce je část programu, která musí být provedena atomicky. Typicky chrání sdílenou proměnnou, registr periferie nebo jiný sdílený prostředek.

Hrubá metoda je dočasné vypnutí přerušení. Tím se zamezí přepnutí kontextu, ale není to efektivní, protože se zastaví i obsluha nesouvisejících výjimek. Elegantnější prostředek je mutex.

Mutex

Mutex znamená mutual exclusion, tedy vzájemné vyloučení. Do chráněné části může vstoupit jen jedno vlákno. Ostatní vlákna jsou blokována, dokud vlastník mutex neuvolní.

  • vstup do kritické sekce: take,
  • opuštění kritické sekce: give,
  • mutex chrání sdílený prostředek,
  • mutex má priority inheritance.

Binární semafor

Binární semafor je podobný mutexu, ale slouží hlavně ke komunikaci a signalizaci mezi vlákny nebo mezi ISR a vláknem.

VlastnostMutexBinární semafor
Typické použitíOchrana kritické sekce.Signalizace události.
Stav po vytvořeníPlný, take neblokuje.Prázdný, take blokuje.
Priority inheritanceAno.Ne.

Čítací semafor

Čítací semafor může mít hodnoty větší než 1. Operace take odečítá 1, operace give přičítá 1.

  • Omezení počtu současných přístupů ke zdroji: semafor se inicializuje hodnotou N.
  • Počítání událostí: generátor volá give, spotřebitel volá take.

Priority inversion a priority inheritance

Priority inversion nastane, když nízkoprioritní úloha drží mutex, vysokoprioritní úloha na něj čeká a mezitím běží úloha se střední prioritou. Priority inheritance tento problém zmírňuje tak, že držitel mutexu dočasně zdědí vyšší prioritu čekající úlohy.

Deadlock

Deadlock vznikne, když se úlohy vzájemně zablokují čekáním na mutexy. Typický příklad: task A drží mutex 1 a čeká na mutex 2, task B drží mutex 2 a čeká na mutex 1.

Rekurzivní mutex řeší situaci, kdy stejné vlákno potřebuje vzít stejný mutex vícekrát. Uvolní se až tehdy, když počet give odpovídá počtu take.

10. Časová základna, SysTick a časovače

SysTick a časová základna

FreeRTOS používá časovou základnu operačního systému. Na ARM Cortex-M je typickým zdrojem této časové základny SysTick.

  • SysTick je 24bitový down-counting timer.
  • Primárně slouží ke generování časové základny OS.
  • Generuje systémovou výjimku.
  • Frekvence ticku se nastavuje přes configTICK_RATE_HZ.
  • Hlavní registry: SYST_CSR, SYST_RVR, SYST_CVR.

Softwarové časovače

Softwarové časovače FreeRTOSu jsou odvozeny od tickové časové základny. Nejsou to hardwarové periferie, ale objekty spravované operačním systémem.
TypPopis
One-shotPo uplynutí periody se zastaví.
PeriodickýPo uplynutí periody se automaticky spustí znovu.

Časovač má handle a může mít také timer ID typu void *. Callback má prototyp:

void callback(TimerHandle_t xTimer)
FunkceVýznam
xTimerCreate()Vytvoří časovač v zastaveném stavu.
xTimerStart()Spustí časovač.
xTimerStop()Zastaví časovač.
xTimerChangePeriod()Změní periodu časovače.
pvTimerGetTimerID()Získá timer ID.

Chyták: callback softwarového časovače neběží v ISR. Volá ho daemon/timer service task. Přesto nesmí blokovat, protože by zablokoval obsluhu dalších timer příkazů a callbacků.

11. Reentrantnost a architektura driveru

Funkce je reentrantní, pokud je navržena tak, aby mohla být bezpečně zavolána vícekrát zároveň, například z různých úloh nebo z úlohy a přerušení.

V prostředí operačního systému je nutné zajistit bezpečné sdílení hardwarových periferií mezi vlákny. Programování periferie obvykle vyžaduje přesné pořadí zápisů do registrů, takže přístup k HW je kritická sekce.

  • FreeRTOS neposkytuje obecné BSP pro konkrétní periferie, programování HW je odpovědnost uživatele.
  • Reentrantnost driveru se typicky zajišťuje mutexem.
  • Vypnutí výjimek nemusí stačit, pokud periferie sama potřebuje IRQ k dokončení operace.

Vzorová architektura driveru

  1. Vlákno získá mutex.
  2. Vlákno zahájí HW akci.
  3. Hardware vyvolá IRQ.
  4. Handler dokončí HW akci a vyčte výsledek.
  5. Handler nastaví event nebo dá semafor.
  6. Vlákno čeká na event.
  7. Vlákno zpracuje výsledek.
  8. Vlákno vrátí mutex.
Příklad: ADC driver. Více úloh může chtít měření, ale jen jedna smí v daném okamžiku programovat ADC registry. Mutex hlídá exkluzivitu, IRQ ukončí převod a event probudí čekající úlohu.

12. SVC, PendSV a kontextové přepnutí

Supervisor Call – SVC

SVC je softwarová výjimka vyvolaná instrukcí SVC #imm. Slouží jako mechanismus pro volání služeb OS, typicky z neprivilegovaného Thread módu do privilegovaného Handler módu.

  • SVC je synchronní výjimka, tedy obsluhuje se ihned.
  • Nemá se spouštět z handleru se stejnou nebo vyšší prioritou, jinak může vzniknout HardFault.

PendSV

PendSV je asynchronní softwarová výjimka určená hlavně pro vnitřní použití operačním systémem. Používá se pro kontextové přepnutí.

  • Vyvolává se nastavením bitu PENDSVSET v registru SCB_ICSR.
  • Je „pendable“, nemusí se provést okamžitě.
  • Typicky má nízkou prioritu, aby se kontext přepínal až po dokončení důležitějších ISR.
Pointa PendSV: ISR může jen říct „po návratu bude potřeba přepnout úlohu“. Samotné těžší přepnutí kontextu se provede později v PendSV, ne přímo v každém přerušení.

Co určitě umět říct u státnic

  • FreeRTOS task je vlákno, ne proces.
  • Scheduler vybírá z ready úloh a respektuje priority.
  • FIFO/Round Robin se řeší mezi úlohami stejné priority.
  • Kooperativní plánování vyžaduje dobrovolné předání CPU, preemptivní ne.
  • Event group signalizuje události, queue přenáší data.
  • Z ISR jen FromISR funkce a žádné blokování.
  • Mutex chrání kritickou sekci a má priority inheritance, semafor slouží hlavně k signalizaci.
  • Timer callback není ISR, běží v daemon tasku a nesmí blokovat.
